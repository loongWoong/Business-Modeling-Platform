## 五、对接ETL和元数据管理可能存在的问题

### 5.1 ETL对接问题

#### 5.1.1 模型到物理表映射缺失
```
问题：当前模型仅有逻辑定义，缺少物理表映射信息

ETL需要的信息：
├── 源表名（physical_table）
├── 字段映射（logical_column → physical_column）
├── 数据类型映射（逻辑类型 → 物理类型）
├── 数据源连接信息
└── 分区策略、增量字段等

建议：
{
  "model": {
    "physicalTable": "t_toll_station",
    "physicalSchema": "toll_system",
    "datasourceId": 1,
    "partitionColumn": "create_date",
    "incrementColumn": "update_time"
  },
  "property": {
    "physicalColumn": "station_id",
    "physicalType": "VARCHAR(32)",
    "jdbcType": "VARCHAR"
  }
}
```

#### 5.1.2 关系约束不足以生成ETL逻辑
```
问题：当前关系只有名称和类型，无法生成JOIN条件

缺失信息：
├── JOIN条件表达式
├── 外键字段对应关系
├── 关联顺序（左连接/右连接/内连接）
└── 过滤条件

建议增加：
{
  "relation": {
    "joinType": "LEFT JOIN",
    "joinCondition": "source.owner_id = target.id",
    "filterCondition": "target.status = 'active'"
  }
}
```

#### 5.1.3 转换规则未定义
```
问题：属性仅有类型定义，无转换规则

ETL常见转换需求：
├── 日期格式转换（yyyy-MM-dd → timestamp）
├── 编码转换（UTF-8 → GBK）
├── 脱敏规则（手机号中间4位*）
├── 派生计算（fullName = firstName + lastName）
└── 数据清洗（去空格、去特殊字符）

建议增加：
{
  "property": {
    "transformRules": [
      {"type": "dateFormat", "pattern": "yyyy-MM-dd HH:mm:ss"},
      {"type": "mask", "rule": "phone_middle_4"}
    ]
  }
}
```

#### 5.1.4 数据源配置与ETL工具不兼容
```
问题：当前数据源配置过于简单

ETL工具（如DataX、Kettle）需要的信息：
├── 完整JDBC连接串
├── 用户名/密码（加密存储）
├── 连接池配置
├── 读写分离配置
├── 分库分表规则
└── 数据采集策略

当前配置：
{ "url": "jdbc:mysql://localhost:3306/db", "tableName": "t_xxx" }

建议配置：
{
  "datasource": {
    "type": "mysql",
    "host": "localhost",
    "port": 3306,
    "database": "expressway",
    "username": "xxx",
    "password": "encrypted:xxx",
    "poolSize": 10,
    "fetchSize": 1000,
    "queryTimeout": 60
  }
}
```

#### 5.1.5 增量同步标识缺失
```
问题：无法确定增量数据范围

增量ETL需要的信息：
├── 增量字段（update_time、version_no）
├── 增量策略（时间戳、版本号、CDC）
├── 增量窗口（每小时、每天）
├── 水位线管理
└── 重跑策略

建议增加：
{
  "model": {
    "incrementStrategy": "timestamp",
    "incrementColumn": "update_time",
    "incrementWindow": "1 HOUR",
    "watermarkTable": "etl_watermark"
  }
}
```

### 5.2 元数据管理对接问题

#### 5.2.1 元数据标准不统一
```
问题：当前元数据格式与业界标准不兼容

常见元数据标准：
├── Apache Atlas（Type/Entity模型）
├── OpenMetadata（JSON-Schema）
├── DataHub（Aspect模型）
└── 自定义Schema

建议：
1. 定义元数据导出接口，支持多种格式
2. 实现与Atlas/DataHub的双向同步
3. 支持元数据变更事件推送
```

#### 5.2.2 血缘信息缺失
```
问题：无法追踪数据来源和流向

元数据管理需要的血缘信息：
├── 字段级血缘（column-level lineage）
├── 表级血缘（table-level lineage）
├── 加工逻辑（transformation logic）
├── 数据源标识
└── 时间版本

建议增加：
{
  "lineage": {
    "sourceColumns": ["source_table.column1"],
    "targetColumn": "target_table.column2",
    "transformLogic": "UPPER(column1)",
    "etlJobId": "job_xxx",
    "effectiveTime": "2025-12-22"
  }
}
```

#### 5.2.3 版本管理与变更追踪不足
```
问题：无法追踪模型变更历史

元数据管理需要：
├── 模型版本历史
├── 变更差异（新增/修改/删除字段）
├── 变更原因和审批记录
├── 变更影响范围
└── 回滚能力

建议增加：
{
  "versionHistory": [
    {
      "version": "1.0.0",
      "changeType": "ADD_COLUMN",
      "changes": [...],
      "reason": "业务需求xxx",
      "approver": "xxx",
      "effectiveDate": "2025-12-22"
    }
  ]
}
```

#### 5.2.4 数据分类分级缺失
```
问题：无法支持敏感数据管理

数据治理需要：
├── 数据分类（个人信息、业务数据、公开数据）
├── 敏感级别（L1-L4）
├── 脱敏规则
├── 访问控制策略
└── 合规要求（GDPR、个保法）

建议增加：
{
  "property": {
    "classification": "PII",
    "sensitivityLevel": "L3",
    "maskRule": "phone_mask",
    "accessPolicy": "need_approval",
    "compliance": ["GDPR", "个保法"]
  }
}
```

#### 5.2.5 数据质量规则缺失
```
问题：无法定义和执行数据质量检查

数据质量管理需要：
├── 完整性规则（NOT NULL、必填）
├── 唯一性规则（主键唯一）
├── 一致性规则（外键存在性）
├── 准确性规则（枚举值范围）
├── 时效性规则（数据新鲜度）
└── 自定义规则（正则表达式、SQL）

建议增加：
{
  "property": {
    "qualityRules": [
      {"type": "not_null", "severity": "error"},
      {"type": "unique", "severity": "error"},
      {"type": "regex", "pattern": "^[A-Z0-9]+$", "severity": "warning"},
      {"type": "range", "min": 0, "max": 100, "severity": "error"}
    ]
  },
  "model": {
    "qualityRules": [
      {"type": "freshness", "maxDelayMinutes": 60},
      {"type": "volume", "minRows": 1000, "maxRows": 10000000}
    ]
  }
}
```

### 5.3 技术集成问题

#### 5.3.1 API集成方式单一
```
问题：仅支持RESTful API，不支持事件驱动

建议增加：
├── Webhook回调（模型变更通知）
├── 消息队列集成（Kafka、RabbitMQ）

├── 批量更新
├── 批量删除

```

#### 5.3.3 权限与审计缺失
```
问题：生产环境需要严格的权限控制

需要增加：
├── 用户认证（OAuth2/LDAP）

├── 数据权限（行级/列级）

```
